# Live board sharing

A **presenter** (an authenticated user in the board app, connected to the JSON-RPC WebSocket
`/ws/v1`) starts a share session and gets an ephemeral 6-character code. Anonymous **viewers** open
the plain-JSON WebSocket `/ws/v1/view/{code}` and receive a cached snapshot plus live relayed delta
messages. The backend may run multi-instance: all session state and fan-out go through Redis, so the
presenter and each viewer can land on different instances.

```text
Presenter (app)                                             Viewers (web)
   │  /ws/v1  sharing.v1.publish                                ▲  /ws/v1/view/{code}
   ▼                                                            │
Instance A ── PUBLISH ──►  Redis channel:{code}  ── relay ──► Instance B (ViewerRegistry)
   ▲                                                            │
   └── notifications ◄──  Redis control:{code}  ◄── joins/pings/resyncs
```

## The server never parses board content

Presenter-published messages are **opaque JSON envelopes**. The server reads only the envelope
fields `type` (string), `seq` (long) and — when `type == "snapshot"` — `fileIds` (string array).
Everything else is stored (snapshots) and relayed (all messages) as the verbatim raw JSON text.
This mirrors the `boards.data` blob rule: the board wire format is owned by the Flutter client and
can evolve without server changes.

## Envelope contract (defined by the client, relayed by the server)

Every presenter message: `{"v":1,"seq":<long>,"type":"<type>", ...opaque payload}`. `seq` increases
monotonically within a session so viewers can discard frames already covered by a later snapshot.
Presenter types (server treats all but `snapshot` identically): `snapshot` (carries `fileIds` +
opaque board payload), `boardProps`, `widgetUpserted`, `widgetsSet`, `strokeProgress`,
`strokeCommitted`, `drawingSet`, `clear`.

### Server-origin frames (server → viewers)

Generated by the server, carrying `"seq":0,"origin":"server"` so viewers can distinguish them from
relayed board content:

```json
{ "v": 1, "seq": 0, "origin": "server", "type": "hello", "state": "live" }
{ "v": 1, "seq": 0, "origin": "server", "type": "sessionPaused" }
{ "v": 1, "seq": 0, "origin": "server", "type": "sessionResumed" }
{ "v": 1, "seq": 0, "origin": "server", "type": "sessionEnded", "reason": "stopped" }
{ "v": 1, "seq": 0, "origin": "server", "type": "viewerCount", "count": 3 }
```

- `hello` — always the first frame after connecting. `state` is one of:
  - `live` — session running; the cached snapshot follows immediately.
  - `waiting` — session exists but is paused and/or has no snapshot yet; content follows when the
    presenter publishes (a cached snapshot, when present, still follows the hello).
  - `notFound` — no session with that code; the socket closes right after.
  - `full` — the session is at `Sharing:MaxViewersPerSession`; the socket closes right after.
- `sessionEnded.reason` — `stopped` (presenter called `sharing.v1.stop`) or `expired` (TTL ran out,
  e.g. the presenter never came back). The socket closes after this frame.

### Viewer → server frames

Anything not listed here — including unparsable JSON — is ignored. Inbound frames are capped at
1 KB; larger frames close the socket (`1009 MessageTooBig`).

```json
{ "type": "ping" }            // presence refresh — send every ~15-30 s
{ "type": "resyncRequest" }   // ask the presenter for a fresh snapshot (rate-limited to 1 per 5 s per socket)
```

## Presenter API

The JSON-RPC methods (`sharing.v1.start/stop/publish/heartbeat`) and the server→presenter
notifications (`sharing.v1.viewerCount`, `sharing.v1.snapshotRequested`, `sharing.v1.ended`) are
documented in [json-rpc-methods.md](json-rpc-methods.md#live-sharing). Key semantics:

- A connection has at most one live session (`RpcConnection.ShareCode`); `sharing.v1.start` is
  idempotent per connection.
- On `sharing.v1.snapshotRequested`, the presenter should publish a fresh `snapshot` envelope. The
  server debounces the notification (250 ms) so a burst of joins triggers one snapshot.
- The presenter heartbeats every ~30 s. Publishing also slides the TTL, so heartbeats only matter
  while idle.

## Lifecycle / state machine

```text
                    sharing.v1.start
                          │
                          ▼
   ┌─────────────────── live ────────────────────┐
   │   publish/heartbeat slide the 180 s TTL     │
   └──┬───────────────────────────────┬──────────┘
      │ presenter WS drops            │ sharing.v1.stop
      ▼                               ▼
    paused ────────────────────► deleted (viewers get sessionEnded{stopped})
      │  ▲                            
      │  └ sharing.v1.start{resumeCode} → live again (viewers get sessionResumed)
      │
      │ TTL expires (no heartbeats while paused)
      ▼
    expired — key vanishes; each instance's viewer watchdog notices within ~10 s
              and sends sessionEnded{expired} to its local viewers
```

- **Pause** (presenter disconnect) keeps the *remaining* TTL as the reconnect grace window — it is
  deliberately not refreshed. Viewers receive `sessionPaused` and should show a "presenter will be
  right back" state.
- **Resume** requires the same `presenterUserId`; it sets the state back to `live`, refreshes the
  TTL, and broadcasts `sessionResumed`. The presenter should follow up with a fresh snapshot.

## Redis keys and channels

All names live in `Services/Sharing/ShareKeys.cs`:

| Name | Type | Content |
| --- | --- | --- |
| `h3xboard:share:session:{CODE}` | hash | `presenterUserId`, `state` (`live`/`paused`), `seq`, `snapshot` (raw JSON), `fileIds` (JSON array), `createdAt` — sliding TTL `Sharing:SessionTtlSeconds` |
| `h3xboard:share:viewers:{CODE}` | zset | member = viewer connection guid, score = unix seconds of last ping; count = entries younger than 45 s, stale ones pruned on read |
| `h3xboard:share:channel:{CODE}` | pub/sub | data channel: presenter envelopes + server-origin frames, relayed verbatim to viewers |
| `h3xboard:share:control:{CODE}` | pub/sub | internal control frames (`__snapshotRequested`, `__viewerCount`) for the instance holding the presenter connection |

Code claiming is an atomic set-if-absent (`Condition.KeyNotExists` transaction); generation retries
up to 5 times on collision (6 chars from a 31-char alphabet ≈ 887M combinations, so collisions are
effectively theoretical).

| Piece | File | Role |

| `SharingRpcV1` | `Rpc/SharingRpcV1.cs` | JSON-RPC surface, registered on `/ws/v1` |
| `ShareSessionService` | `Services/Sharing/ShareSessionService.cs` | Session lifecycle + publish hot path (scoped per presenter connection) |
| `PresenterNotifier` | `Services/Sharing/PresenterNotifier.cs` | Control-channel → debounced JSON-RPC notifications on the presenter connection |
| `RpcConnection` | `Rpc/RpcConnection.cs` | Scoped holder of the connection's `JsonRpc` instance + active share code |
| `IShareStore` | `Services/Sharing/IShareStore.cs` | Session hash + viewer presence (`RedisShareStore` / `InMemoryShareStore`) |
| `IShareBus` | `Services/Sharing/IShareBus.cs` | Per-code pub/sub, refcounted per instance (`RedisShareBus` / `InMemoryShareBus`) |
| `ViewerRegistry` | `Services/Sharing/ViewerRegistry.cs` | Singleton: code → local viewer queues; fan-out + slow-viewer drop + expiry watchdog |
| Viewer WS | `Rpc/ViewWsEndpoints.cs` | `/ws/v1/view/{code}` — anonymous, plain JSON |
| View files | `Rest/ViewFileEndpoints.cs` | `GET /api/v1/view/{code}/files/{fileId}` — anonymous, snapshot-scoped |
| Rate limiter | `Services/Sharing/ShareCodeRateLimiter.cs` | Per-IP token bucket (10/min) on viewer code lookups |

**Backpressure**: each local viewer gets a bounded queue (256 frames). A viewer that cannot drain
it is dropped (`1011` close) rather than stalling the relay — it can reconnect and resync.

**Cross-instance presenter events**: viewer joins/pings/resyncs are published as internal control
frames on `h3xboard:share:control:{CODE}`; the instance holding the presenter connection subscribes
and forwards them as debounced notifications. Nothing assumes the presenter and viewer share an
instance.

## File downloads for viewers

`GET /api/v1/view/{code}/files/{fileId}` streams a file (e.g. a board background) **only** when
`fileId` is listed in the session's latest snapshot `fileIds`, acting on behalf of the presenter
through the same owner-scoped `FileService` path as the authenticated download endpoint. Responses
carry `Cache-Control: private, max-age=3600`. Unknown code or unlisted file are both a plain 404.

## Security notes

- The share code is the only credential for viewing — hence the crypto RNG, the ambiguity-free
  alphabet, and the per-IP token bucket (10 lookups/min) on the viewer endpoint. The limiter is
  per-instance and in-memory by design; multiply by instance count when reasoning about it.
- The viewer endpoint deliberately has **no** session auth gate (unlike `/ws/v1`), reads at most
  1 KB frames, and understands exactly two frame types.
- Viewers never send anything to the presenter or the board — the relay is strictly one-way; the
  only viewer-triggered effects are presence updates and a (rate-limited, debounced) snapshot
  request.

## Configuration

| Key | Default | Meaning |
| --- | --- | --- |
| `Sharing:SessionTtlSeconds` | `180` | Sliding session TTL; also the presenter-reconnect grace window while paused |
| `Sharing:MaxViewersPerSession` | `100` | Join cap per session — extra viewers get `hello {state:"full"}` |
| `Sharing:MaxMessageBytes` | `524288` | Max total size of one `sharing.v1.publish` batch |
| `Server:WebAppUrl` | *(unset)* | Public URL of the web viewer app, surfaced as `webAppUrl` in `/api/v1/server/info` |

Redis vs in-memory follows the existing `Redis:ConnectionString` switch in `Program.cs`: when it is
empty, sharing falls back to in-process store/bus implementations (single instance only, sessions
lost on restart) and a startup warning is logged — mirroring the session-cache fallback.
